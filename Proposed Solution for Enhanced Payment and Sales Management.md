# Proposed Solution for Enhanced Payment and Sales Management

This document outlines a comprehensive solution to address the current challenges in payment processing, POS integration, and credit sales management within the SabiOPS system. The goal is to provide accurate record-keeping, improve reconciliation, and enhance the flexibility of handling various payment scenarios for Nigerian SMEs.

## 1. Enhanced Payment Tracking and POS Integration

### Problem Statement:
Nigerian SMEs often use a combination of POS and cash transactions. The current system lacks the ability to automatically track money coming in and going out through POS, differentiate between card and bank transfer payments, and calculate daily cash-at-hand and total POS deposits/withdrawals.

### Proposed Solution:

#### 1.1. Payment Method Standardization and Expansion:

To accurately track different payment channels, the `payment_method` field in both `payments` and `sales` tables needs to be standardized and expanded. Instead of free-form text, we should introduce an `ENUM` type or a lookup table for `payment_methods` to ensure consistency and prevent data entry errors. This will allow for clear categorization of transactions (e.g., 'Cash', 'POS_Card', 'POS_Transfer', 'Bank_Transfer', 'Credit').

**Database Schema Changes (Conceptual):**

*   **`payment_methods` (New Lookup Table):**
    *   `id` (Primary Key, UUID)
    *   `name` (e.g., 'Cash', 'POS - Card', 'POS - Transfer', 'Bank Transfer', 'Credit', 'Online Payment')
    *   `type` (e.g., 'Cash', 'Digital', 'Credit') - This can help in categorizing for reporting.

*   **`payments` and `sales` tables:**
    *   Change `payment_method` column type to reference `payment_methods.id` (Foreign Key).

#### 1.2. Detailed POS Transaction Tracking (Manual Input Focus):

To accurately track POS transactions (deposits and withdrawals) without direct terminal integration, the system will rely on structured manual input. This involves enhancing the `payments` table to capture necessary details.

**Enhance `payments` table:**

*   Add new columns to the `payments` table to capture details relevant to POS transactions:
    *   `is_pos_transaction` (Boolean, default FALSE): To flag if the payment was processed via a POS system (manual selection by user).
    *   `pos_account_name` (Text, nullable): To record the name of the POS account (e.g., 'Moniepoint POS', 'Opay POS') for reconciliation.
    *   `transaction_type` (ENUM: 'Sale', 'Refund', 'Deposit', 'Withdrawal'): To differentiate between money coming in (sales/deposits) and going out (refunds/withdrawals) via POS. This will be manually selected by the user.
    *   `pos_reference_number` (Text, nullable): To store the reference number generated by the physical POS terminal for reconciliation purposes.

This approach allows the user to manually categorize and detail POS transactions, providing the necessary data for accurate daily summaries without requiring direct API integrations with various POS terminal providers.

#### 1.3. Manual Recording of Diverse Payment Types:

Given that many Nigerian SMEs utilize various POS terminals (Moniepoint, OPay, PalmPay, etc.) that may not offer direct API integration, the system will focus on providing a robust and user-friendly mechanism for *manual recording* of diverse payment types. This ensures flexibility and broad applicability for businesses using any payment method.

*   **Manual Payment Recording (`record_payment` function in `payment.py`):**
    *   The `payment_method` field in the request payload will be crucial. Instead of relying on automatic detection from payment gateways, the user will explicitly select or input the payment method for each transaction.
    *   The system will use a standardized set of payment methods (e.g., 'Cash', 'POS - Card', 'POS - Transfer', 'Bank Transfer', 'Credit', 'Online Payment - Paystack', 'Online Payment - Other').
    *   For POS transactions, the user will select 'POS - Card' or 'POS - Transfer' as appropriate. This allows for accurate categorization of funds received through different POS channels, even without direct integration.
    *   The `payment.py` module will be updated to handle these distinct payment methods, ensuring that each transaction is correctly tagged.

**Implementation Steps for Manual Recording:**
1.  Ensure the `payment_methods` lookup table (as described in 1.1) is comprehensive and includes all relevant manual payment options (e.g., 'POS - Card', 'POS - Transfer', 'Bank Transfer', 'Cash', 'Credit').
2.  Modify the `record_payment` function in `payment.py` to accept and validate the selected `payment_method_id` (or a string that can be mapped to it) from the frontend. This function will be the primary entry point for all payment records, whether from online gateways or manual input.
3.  Update the frontend forms to provide clear options for users to select the payment method for each sale or payment record. This will be a dropdown or radio button selection based on the `payment_methods` lookup table.

#### 1.4. Daily Financial Summaries (Cash at Hand, POS Totals, Product Category Sales):

This requires new API endpoints and logic to aggregate data from the `payments` and `sales` tables.

*   **New API Endpoint (`/api/reports/daily_summary`):**
    *   **Cash at Hand:** Calculate by summing all 'Cash' payments (money in) and subtracting all 'Cash' expenses/withdrawals (money out) for a given day.
        *   This requires a robust `transactions` table that clearly marks money in/out and links to payment methods.
        *   Ensure all cash-related expenses (e.g., petty cash withdrawals) are recorded in the `expenses` or `transactions` table with a 'Cash' payment method.
    *   **Total Money Deposited on POS:** Sum of all `amount` from `payments` where `is_pos_transaction` is TRUE and `transaction_type` is 'Sale' or 'Deposit' for the day.
    *   **Total Money Withdrawn from POS:** Sum of all `amount` from `payments` where `is_pos_transaction` is TRUE and `transaction_type` is 'Withdrawal' or 'Refund' for the day.
    *   **Total Amount from Drinks (Product Category):**
        *   Requires a `product_categories` table and linking `products` to `product_categories`.
        *   Modify `sales` table to include `product_category_id` or join with `products` table to get category.
        *   Aggregate `total_amount` from `sales` for products belonging to the 'Drinks' category for the day.

**Implementation Steps:**
1.  Create `payment_methods` lookup table and populate with standard methods, including manual POS options (e.g., 'POS - Card', 'POS - Transfer').
2.  Migrate `payments` and `sales` tables to use `payment_method_id` (Foreign Key).
3.  Add `is_pos_transaction`, `pos_account_name`, `transaction_type`, and `pos_reference_number` to the `payments` table.
4.  Modify `record_payment` in `payment.py` to accept and validate `payment_method_id`, `is_pos_transaction`, `pos_account_name`, `transaction_type`, and `pos_reference_number` from user input.
5.  Ensure all money-out transactions (expenses, refunds) are correctly recorded with their payment methods.
6.  Create `product_categories` table and link `products` to categories.
7.  Develop new API endpoint (`/api/reports/daily_summary`) to generate the required daily summaries based on the manually recorded data.

## 2. Invoice Status Update and Credit Sales Management

### Problem Statement:
Users are encountering errors when updating invoice status (`backend\sabiops-backend\migrations\Existing.sql`). Payment methods are inconsistent, and there's a need for a robust credit sales system that allows partial payments, accurate revenue recognition, and tracking of overdue amounts.

### Proposed Solution:

#### 2.1. Resolving Invoice Status Update Error:

The error in `Existing.sql` when updating invoice status suggests a potential issue with the migration script itself, or a conflict with existing data or constraints. The `invoice` table schema shows `status` as `text` with a default of `'draft'::text`. The `update_invoice` function in `invoice.py` prevents updating a 'paid' invoice, but doesn't explicitly handle other status transitions or potential database triggers/constraints.

**Investigation Steps:**
1.  **Review `Existing.sql`:** Carefully examine the `Existing.sql` file for any `ALTER TABLE` statements, `TRIGGER` definitions, or `CONSTRAINT` definitions related to the `invoices` table, especially those affecting the `status` column or any related foreign keys.
2.  **Check Database Logs:** When the error occurs, capture the exact database error message. This will provide clues about constraint violations (e.g., foreign key, check constraint) or data type mismatches.
3.  **Supabase Console:** Inspect the `invoices` table directly in the Supabase console for any custom policies, triggers, or RLS (Row Level Security) rules that might be interfering with updates.
4.  **Code Review (`invoice.py`):** Ensure that the `update_invoice` function correctly forms the `UPDATE` query and that the `status` value being sent is valid according to any database constraints or expected enum values.

**Common Causes & Fixes:**
*   **Constraint Violation:** If there's a `CHECK` constraint on the `status` column allowing only specific values, ensure the update is sending one of those values. If it's a foreign key constraint, ensure the linked record exists.
*   **Trigger Error:** A database trigger on the `invoices` table might be failing. The trigger's code needs to be reviewed.
*   **RLS Policy:** An RLS policy might be preventing the update. Test with a user having full permissions to rule this out.
*   **Data Type Mismatch:** Although `status` is `text`, ensure no implicit type conversions are failing.

#### 2.2. Consistent Payment Methods and Sale Status Update:

The current `payment_method` inconsistency and the desire to update `sale` status from 'credit'/'pending' to 'cash'/'online payment' require a unified approach.

**Proposed Changes:**

1.  **Unified Payment Status:** Introduce a clear `payment_status` column in the `sales` table (already exists as `payment_status` with default `'completed'::text`). This should reflect the payment state of the *sale*, distinct from the `payment_method`.
    *   **`payment_status` ENUM/Values:** 'Pending', 'Partially Paid', 'Paid', 'Credit', 'Cancelled'.

2.  **Flexible Sale Status Update Endpoint (`/api/sales/<sale_id>/update_status`):**
    *   Create a new endpoint or enhance the existing `update_sale` (if it exists) to specifically handle status and payment updates for sales.
    *   This endpoint will allow an admin/salesperson to change the `payment_status` of a sale.
    *   **Logic:**
        *   If `payment_status` changes from 'Credit' or 'Pending' to 'Paid' (or a specific payment method like 'Cash', 'POS_Card'), the sale's revenue and profit should then be recorded.
        *   The `payment_method` for the sale should be updated to reflect how the payment was eventually settled (e.g., 'Cash', 'POS_Card').

3.  **Partial Payments for Credit Sales:**
    *   **`sales` table enhancement:** Add `amount_paid` (Numeric, default 0) and `amount_due` (Numeric, default `total_amount`) columns to the `sales` table.
    *   **New `sale_payments` table (for detailed tracking):**
        *   `id` (Primary Key, UUID)
        *   `sale_id` (UUID, Foreign Key to `sales`)
        *   `payment_id` (UUID, Foreign Key to `payments` - if a formal payment record is created)
        *   `amount_paid` (Numeric)
        *   `payment_date` (Timestamp with Time Zone)
        *   `payment_method_id` (UUID, Foreign Key to `payment_methods`)
        *   `notes` (Text, nullable)
        *   `created_at`, `updated_at`

    *   **Update Logic for Partial Payments:**
        *   When a partial payment is made for a credit sale, a new record is added to `sale_payments`.
        *   The `amount_paid` in the `sales` table is incremented by the partial payment amount.
        *   The `amount_due` in the `sales` table is decremented.
        *   If `amount_due` becomes 0, the `payment_status` of the sale changes to 'Paid'.
        *   Revenue and profit for the sale are recognized *only when the `payment_status` becomes 'Paid'*.

#### 2.3. Revenue and Profit Recognition for Credit Sales:

*   **Current Issue:** Credit sales are recorded on total revenue/profit even if not paid.
*   **Solution:** Modify the revenue and profit calculation logic (likely in `sales.py` and any reporting modules) to *exclude* sales with `payment_status` as 'Credit' or 'Pending' from `total_revenue` and `gross_profit` calculations until their `payment_status` changes to 'Paid'.
*   **Reporting:** Introduce a separate metric for 'Outstanding Credit Sales' or 'Accounts Receivable' to track the value of unpaid credit sales.

**Implementation Steps:**
1.  **Database Migrations:**
    *   Add `amount_paid` and `amount_due` to `sales` table.
    *   Create `sale_payments` table.
    *   Ensure `payment_status` in `sales` table is correctly defined and used.
2.  **Backend Logic (`sales.py`, `payment.py`):**
    *   Modify `create_sale` to set initial `amount_paid` to 0 and `amount_due` to `total_amount` for credit sales.
    *   Implement the new endpoint for updating sale status and handling partial payments.
    *   Update `record_payment` to link payments to sales and update `amount_paid`/`amount_due` in `sales` table.
3.  **Reporting/Analytics (`analytics.py`, `dashboard.py`):**
    *   Adjust queries for `total_sales`, `gross_profit`, and `profit_from_sales` to filter by `payment_status = 'Paid'`.
    *   Add new queries for 'Outstanding Credit Sales'.

## 3. HTML Download Format for Daily Summary

### Problem Statement:
There is an existing HTML download format in the `utils` folder, and the user wants the daily summary (cash at hand, POS totals, product category sales) to be downloadable in this format.

### Proposed Solution:

1.  **Locate Existing HTML Template:** Identify the specific HTML template file in the `utils` folder (e.g., `utils/download_format.html`).
2.  **Data Preparation:** The new `/api/reports/daily_summary` endpoint will provide the necessary data (cash at hand, POS totals, drinks sales).
3.  **Templating Engine:** If not already in use, consider a simple templating engine (like Jinja2 if Flask is used, or a custom string formatter) to inject the dynamic data into the HTML template.
4.  **New Endpoint for HTML Download:** Create a new endpoint (e.g., `/api/reports/daily_summary/download_html`) that:
    *   Calls the `/api/reports/daily_summary` endpoint to get the data.
    *   Loads the HTML template.
    *   Renders the template with the fetched data.
    *   Returns the rendered HTML file with appropriate headers (`Content-Disposition: attachment; filename="daily_summary.html"`).

**Implementation Steps:**
1.  Verify the path and structure of the existing HTML download format file.
2.  Implement the data fetching logic from the daily summary endpoint.
3.  Integrate a templating mechanism to populate the HTML.
4.  Create the download endpoint.
